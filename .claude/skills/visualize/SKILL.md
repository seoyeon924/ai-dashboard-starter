# Visualize Skill

## 목적
"예쁜 시각화"가 아니라 "3초 안에 핵심이 읽히는 시각화"를 만든다.
**독자 정의 → 와이어프레임 → 디자인 순서를 반드시 지킨다.**

## 이 스킬이 하는 일
- 독자(의사결정자) 정의
- 와이어프레임 텍스트로 먼저 기획
- 질문에 맞는 차트 선택 + 근거 제시
- DataBridge 디자인 원칙 적용하여 HTML 대시보드 제작
- **완성 후 /review로 전달**

## 차트 선택 원칙
- 추세를 보여줄 때만 line 차트를 쓴다.
- 범주 비교는 정렬된 bar를 우선 검토한다.
- 불확실성이 중요하면 CI 또는 error bar 포함 차트를 제안한다.
- 분포를 보여주려면 scatter, histogram, box plot을 고려한다.
- 복잡한 차트는 해석 비용보다 얻는 가치가 클 때만 쓴다.

## 레이아웃 원칙
1. 상단: 핵심 KPI (질문 형태 카드 제목 필수)
2. 중단: 변화 추세 또는 비교
3. 하단: 세그먼트 / 상세 분석 / 액션 포인트

## DataBridge 디자인 원칙 (필수 적용)
- 배경: `#f5f5f5`
- 카드: 흰 배경 `#ffffff`, 테두리 `#e8e8e8`
- 사이드바: `#111111` (검정)
- KPI 수치: `#0a0a0a` 검정 고정 — 빨강/초록 금지
- 포인트: 빨강 `#ef7d86` (위험/나쁨), 초록 `#35c995` (안전/좋음)
- 카드 제목: 질문 형태 ("어디가 문제인가?")
- STEP 레이블 구조 사용
- 폰트: Inter 또는 Pretendard
- 차트: 순수 SVG/Canvas 직접 그리기 (CDN 렌더링 이슈 방지)

## 기존 대시보드 참조 원칙
새 대시보드 제작 시 기존 파일을 먼저 확인하고 구조를 재사용한다.
- `analysis-dashboard.html` — 사이드바 + 인터랙티브 KPI 레이아웃 참조
- `dashboard.html` — 7단계 파이프라인 모니터 레이아웃 참조
- `dashboards/dash-sales.html`, `dash-hr.html` 등 — 도메인별 패턴 참조

## 출력 형식

### 1. 독자 정의
누가 보는 대시보드인가? 이 사람이 내려야 하는 결정은?

### 2. 와이어프레임 (텍스트)
```
[상단] KPI 카드 N개
[중단 좌] [차트 타입]: [목적]
[중단 우] [차트 타입]: [목적]
[하단] [테이블 또는 AI 인사이트]
```

### 3. 차트별 선택 근거
- 차트 타입 / 선택 이유 / 오독 위험

### 4. HTML 대시보드 제작 + 브라우저 링크 안내

### 5. 다음 단계
> `/review "[대시보드 파일 경로]"` 를 실행하면 최종 검토를 시작합니다.

## 금지
- 이유 없는 3D 차트 / 복잡한 범례 남발
- 색만으로 의미 전달 / 한 화면에 차트 8개 이상
- 장식용 그라디언트 / KPI 수치에 컬러 사용
- **한쪽 변만 있는 border** (border-left, border-top 등 단독 사용 금지)
